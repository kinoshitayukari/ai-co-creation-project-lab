はじめに

金融データ分析は本質的に「時系列分析」である。価格、出来高、金利、為替レートはすべて時間順に並んだデータであり、その性質を正しく理解しなければ、信頼できる分析結果は得られない。

このカリキュラムでは、『Pythonによる時系列分析 予測モデル構築と企業事例』『Pythonによる時系列予測』などの体系的知識を基に、金融データ特有の時系列分析手法を学ぶ。

さらに、金融時系列で最も危険な落とし穴である先読みバイアス（Look-ahead Bias）について、具体的な発生パターンと回避方法を徹底的に解説する。

注意事項

このカリキュラムは、投資助言や売買推奨を目的としたものではありません。金融データの時系列分析手法の学習を目的とした教育コンテンツです。

第1部: 時系列分析の基礎

第1章: 時系列データとは何か

目的: 金融データが持つ時系列としての性質を理解する。

1.1 時系列データの定義

  • 一定の時間間隔で観測されたデータの系列
  • 金融データの例: OHLCV（始値・高値・安値・終値・出来高）
  • 時間の順序が意味を持つ（シャッフルすると情報が失われる）

1.2 時系列データの7つの特徴量

  • トレンド（上昇・下降・横ばい）
  • 季節性（曜日効果、月末効果、四半期効果）
  • 周期性（景気循環など不規則な周期）
  • ノイズ（ランダムな変動）
  • レベルシフト（構造変化）
  • 外れ値（フラッシュクラッシュなど）
  • 自己相関（過去の値との相関）

1.3 金融時系列の特殊性

通常の時系列データ（気温、売上など）と金融データの違い:

  • 非定常性が強い（平均・分散が時間とともに変化する）
  • ファットテール（正規分布より極端な値が出やすい）
  • ボラティリティクラスタリング（変動の大きい期間が固まる）
  • 効率的市場仮説（予測が困難な理由の理論的背景）
  • レジームチェンジ（市場の性質が突然変わる）

AIエージェント活用法:

プロンプト例:
「BTCUSDの日足終値データについて、以下を分析してください。
1. ADF検定で定常性を確認
2. ACF/PACFプロットを作成
3. リターンの分布をヒストグラムで表示し、正規分布と重ねて比較
4. ローリング平均とローリング標準偏差を30日窓で表示」

第2章: 時系列データの前処理

目的: 分析・モデル構築に適した形にデータを整える。

2.1 欠損値の処理

  • 前方埋め（Forward Fill）: 金融データの標準的手法
  • 線形補間: 短い欠損区間に有効
  • 削除: 欠損が多い期間は除外する判断も必要

2.2 リサンプリング

  • ダウンサンプリング: M1 → H1 → Daily → Weekly
  • OHLCV変換ルール: Open=first, High=max, Low=min, Close=last, Volume=sum
  • アップサンプリングの危険性（存在しないデータを作ることになる）

2.3 定常化

  • 差分変換（1次差分、対数差分）
  • 対数変換
  • 季節差分

2.4 特徴量エンジニアリング

  • リターン（単純リターン、対数リターン）
  • 移動平均（SMA、EMA、WMA）
  • ボラティリティ（ヒストリカル、ローリング）
  • テクニカル指標（RSI、MACD、ボリンジャーバンド）
  • ラグ特徴量（過去N日のリターンを特徴量にする）

AIエージェント活用法:

プロンプト例:
「BTCUSDのM1データをH1にリサンプリングしてください。
OHLCVの変換ルールを正しく適用し、
さらに20期間SMA、RSI(14)、MACD(12,26,9)を追加してください。
欠損値はforward fillで処理。」

第3章: 統計的時系列モデル

目的: 古典的な時系列予測モデルの仕組みと限界を理解する。

3.1 移動平均モデル（MA）

  • ホワイトノイズの加重和としての定式化
  • 次数の決定（ACFプロット）

3.2 自己回帰モデル（AR）

  • 過去の値からの線形予測
  • 次数の決定（PACFプロット）

3.3 ARMA / ARIMA

  • ARとMAの組み合わせ
  • 差分操作による非定常データへの対応（I: Integrated）
  • AIC / BICによるモデル選択

3.4 SARIMA / SARIMAX

  • 季節性の取り込み（Seasonal）
  • 外生変数の追加（X: eXogenous）
  • 金融データでの使用例: 曜日効果、月末効果の取り込み

3.5 Prophet

  • Facebookが開発した時系列予測ライブラリ
  • トレンド + 季節性 + イベント効果の分解
  • 金融データへの適用と限界

3.6 VAR（ベクトル自己回帰）

  • 複数の時系列を同時にモデル化
  • グレンジャー因果性検定
  • 金融データでの使用例: 金利と株価の相互影響

AIエージェント活用法:

プロンプト例:
「BTCUSDの日足終値について、以下を実行してください。
1. ADF検定で定常性を確認し、必要なら差分を取る
2. ACF/PACFからARIMAの次数を推定する
3. auto_arimaでモデルを自動選択する
4. 過去30日を除外して学習し、30日先の予測を行う
5. 予測結果と実績を重ねてプロットする」

第4章: 機械学習による時系列予測

目的: 統計モデルを超えた手法の可能性と限界を理解する。

4.1 時系列データの学習・テスト分割

  • 通常のランダム分割は使えない（時間順序の保持が必須）
  • 時系列分割: 過去で学習 → 未来で検証
  • ウォークフォワード検証

4.2 勾配ブースティング（XGBoost / LightGBM）

  • ラグ特徴量を使ったテーブルデータ化
  • テクニカル指標を特徴量に変換
  • 重要度分析による特徴量の選択

4.3 LSTM / GRU

  • 時系列データのシーケンスとしての入力
  • ウィンドウサイズの設定
  • 過学習の検出と対策

4.4 Transformer / Temporal Fusion Transformer

  • アテンション機構による長期依存関係の学習
  • 解釈可能な予測モデル

4.5 予測の評価指標

  • MAE / RMSE / MAPE
  • 方向一致率（上昇/下降の的中率）
  • 予測モデルの限界を正直に伝える重要性

第2部: 先読みバイアス完全ガイド

第5章: 先読みバイアスとは何か

目的: 金融分析で最も危険な落とし穴を完全に理解する。

5.1 先読みバイアスの定義

先読みバイアス（Look-ahead Bias）とは、バックテストや分析の時点では知り得なかった「未来の情報」を使ってしまうことで、結果が実際よりも良く見えてしまう問題。

この問題があると:

  • バックテストでは素晴らしい結果が出る
  • 実際の運用では全く機能しない
  • 見つけるのが極めて難しい（結果が良いので疑わない）

5.2 先読みバイアスが発生する主な原因

原因 具体例 危険度
同一バーでのシグナルとエントリー 終値でシグナルを出し、同じ終値で売買する 高
特徴量の計算ミス 移動平均の計算に未来のデータを含める 高
異なる頻度のデータ結合 当日の日足確定値を日中に使う 非常に高
テクニカル指標の誤用 一目均衡表の遅行スパンの扱いを間違える 高
正規化・スケーリング テスト期間を含めた全期間でmin-max正規化する 高
経済指標の発表タイミング GDPを発表前の日付で使う 中
銘柄の生存者バイアス 上場廃止銘柄を除外したデータで検証する 中

第6章: 異なる時系列データの結合における先読み

目的: 複数のデータソースを安全に組み合わせる方法を学ぶ。

6.1 頻度の異なるデータの結合

最も見落としやすい先読みの温床。

NG例: 日足データを1時間足に結合する

# NG: 当日の日足終値を日中のデータに結合してしまう
# 10:00の時点で、まだ確定していない日足終値を使っている
df_h1['daily_close'] = df_h1['date'].map(daily_close_dict)

OK例: 前日の日足データのみを使う

# OK: 前日までに確定したデータのみを結合
df_h1['prev_daily_close'] = df_h1['date'].map(
    lambda d: daily_close_dict.get(d - timedelta(days=1))
)

6.2 Point-in-Time（時点整合性）の原則

すべてのデータ結合において守るべき鉄則:

「その時刻に、その情報は本当に利用可能だったか？」

具体的なチェックリスト:

  • [ ] 日足の指標を日中データに使うとき、前日の確定値だけを使っているか
  • [ ] 経済指標（GDP、雇用統計等）を使うとき、発表日時以降のデータにのみ含めているか
  • [ ] 週足・月足のデータを日足に結合するとき、前週・前月の確定値を使っているか
  • [ ] 他の銘柄や資産のデータを結合するとき、同時刻の情報か確認したか
  • [ ] タイムゾーンの変換は正しく行われているか（UTC ↔ JST）

6.3 結合後の検証方法

AIエージェント活用法:

プロンプト例:
「以下のデータ結合コードに先読みバイアスがないか検証してください。
具体的に確認してほしい点:
1. 日足SMAをH1データに結合する箇所で、当日の未確定データが混入していないか
2. merge後のDataFrameで、各行の時刻よりも未来のデータが含まれていないか
3. forward fillが未来方向に作用していないか
検証用のassert文も書いてください。」

第7章: テクニカル指標における先読みバイアス

目的: 一目均衡表をはじめとするテクニカル指標で発生する先読みを理解する。

7.1 一目均衡表の先読みリスク

一目均衡表は5つの線で構成されるが、その中に先読みが発生しやすい要素が含まれている。

線 計算方法 先読みリスク
転換線 過去9期間の(高値+安値)/2 なし
基準線 過去26期間の(高値+安値)/2 なし
先行スパン1 (転換線+基準線)/2を26期間先にプロット あり（後述）
先行スパン2 過去52期間の(高値+安値)/2を26期間先にプロット あり（後述）
遅行スパン 終値を26期間過去にプロット あり（後述）

先行スパン1・2の注意点

先行スパンは「26期間先にプロット」する。チャート上では未来に表示されるが、計算に使うのは過去のデータ。

バックテストでの問題:

NG: 「現在の価格が雲（先行スパン）の上にあるか」を判定するとき、
    未来にプロットされた雲の位置を、現在の価格と比較してしまう。

OK: 「26期間前に計算された雲の値」と「現在の価格」を比較する。
    つまり、26期間前の先行スパンの値が、今日の時点で利用可能な雲の位置。

遅行スパンの注意点

遅行スパンは「終値を26期間過去にプロット」する。

バックテストでの問題:

NG: 遅行スパンが価格を上抜けたことをシグナルにする場合、
    「今日の終値」を「26日前の位置」にプロットして比較するため、
    26日前の時点では「今日の終値」は知り得ない。

OK: 遅行スパンのシグナルは「今日の終値が確定した時点」で初めて発生する。
    エントリーは翌日の始値以降でなければならない。

7.2 その他のテクニカル指標の先読みリスク

ボリンジャーバンド

NG: バンド幅の計算に将来のデータを含むローリングウィンドウを使う
OK: ローリングウィンドウは過去方向のみ（min_periods設定を忘れない）

ピボットポイント

NG: 当日のピボットポイントを当日の日中取引に使う
    （ピボットポイントは前日のHLC（高値・安値・終値）から計算するが、
    当日の日足データで計算してしまうケース）
OK: 必ず前日の確定データから計算する

ATR（Average True Range）

NG: ATR計算のウィンドウに当日の高値・安値を含める
OK: ATRは前日までのデータで計算し、当日のポジションサイズ計算に使う

第8章: バックテストにおける先読みバイアスの体系的回避

目的: 先読みを防ぐための具体的なルールと検証方法を身につける。

8.1 エントリー・エグジットのタイミングルール

ルール 説明
シグナル確定 シグナルはバーの確定後（終値が決まった後）に判定する
エントリー遅延 シグナル確定の次のバーの始値以降でエントリーする
同一バー禁止 シグナル発生バーの終値でエントリーしない
コスト反映 スプレッド、手数料、スリッページを必ず加算する

8.2 特徴量計算のルール

ルール 説明
ローリングウィンドウ 過去方向のみ。min_periodsを設定する
正規化 学習期間のみで計算した統計量でテスト期間も正規化する
ラベル作成 shift(-1)で翌日リターンを計算する場合、最終行がNaNになることを確認
forward fill 過去の値を現在に持ち越す方向のみ使用する

8.3 ウォークフォワード検証

期間1: [学習: 2020-01〜2022-12] → [検証: 2023-01〜2023-06]
期間2: [学習: 2020-07〜2023-06] → [検証: 2023-07〜2023-12]
期間3: [学習: 2021-01〜2023-12] → [検証: 2024-01〜2024-06]
...

重要: 各期間で使えるのは、検証開始日より前のデータのみ。

8.4 先読みバイアス検出のためのユニットテスト

AIエージェント活用法:

プロンプト例:
「以下のバックテストコードに対して、先読みバイアスを検出する
ユニットテストを書いてください。具体的に検証する項目:
1. シグナル発生日とエントリー日が同一でないこと
2. 特徴量の計算に使うウィンドウが過去方向のみであること
3. テスト期間のデータが学習に混入していないこと
4. merge操作で未来の行のデータが過去に漏れていないこと
pytestで実行できる形式で出力してください。」

8.5 先読みバイアス チェックシート

分析やバックテストを完了した後に、必ず確認する項目:

  • [ ] シグナルは足の確定後に判定しているか
  • [ ] エントリーはシグナル確定の次の足以降か
  • [ ] 移動平均・RSI等の計算に未来のデータが入っていないか
  • [ ] 一目均衡表の雲は26期間前の計算値を使っているか
  • [ ] 遅行スパンのシグナルは終値確定後に判定しているか
  • [ ] 異なる頻度のデータ結合で、未確定データを使っていないか
  • [ ] 経済指標は発表日時以降のデータにのみ含まれているか
  • [ ] 正規化・スケーリングは学習期間のみの統計量で行っているか
  • [ ] ウォークフォワードで検証期間のデータが学習に混入していないか
  • [ ] 結合後のDataFrameで、各行より未来のデータが含まれていないか

まとめ

金融データ分析で最も大切なのは、「結果が良い」ことではなく「結果が正しい」ことである。

先読みバイアスが入った分析は、どんなに優れた結果が出ても無価値。むしろ、誤った自信を与えるため有害ですらある。

AIエージェントを活用する最大のメリットは、コードの自動生成だけでなく、「このコードに先読みバイアスはないか？」と質問できることにある。

分析の型を正しく身につけ、AIに検証を依頼し、信頼できる結果だけを残す。これが、金融時系列分析の基本姿勢である。
